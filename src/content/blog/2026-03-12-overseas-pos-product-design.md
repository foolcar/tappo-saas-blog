## 香港茶餐廳：全世界最複雜的點餐場景

如果你能在香港茶餐廳的POS系統做好，你就能在任何市場做好餐飲POS。

香港茶餐廳的點餐複雜度是驚人的：
- 「凍檸茶走甜少冰」——6個字包含4個修改指令
- 「A餐轉B餐個湯，加個菠蘿油，走包」——跨套餐修改
- 「例湯轉羅宋湯，意粉轉飯，加底」——深度客製化

這些在歐美快餐場景中幾乎不存在。麥當勞的POS只需要處理「要不要加大」、「飲品選什麼」這樣的簡單選項。

## POS系統的核心架構設計

一個能跨市場使用的POS，其核心架構必須是「全球化骨架 + 本地化血肉」：

```
┌────────────────────────────────────────┐
│            全球核心模塊                  │
│   訂單引擎 │ 結帳引擎 │ 廚房顯示 │ 報表  │
└────────────────────────────────────────┘
                    │
┌────────────────────────────────────────┐
│           可配置本地化層                 │
│  菜單結構 │ 修改器 │ 稅率 │ 支付 │ 語言 │
└────────────────────────────────────────┘
```

### 菜單修改器（Modifier）架構

這是最難設計的部分。香港茶餐廳需要深度嵌套的修改器，但美國漢堡店只需要簡單的加減配料。

我們的解決方案：**修改器分級系統**

```typescript
interface ModifierGroup {
  id: string;
  name: string;           // "飲品選項"
  type: 'single' | 'multi' | 'nested';
  minSelect: number;      // 最少選擇數量
  maxSelect: number;      // 最多選擇數量 (0=無限制)
  required: boolean;
  modifiers: Modifier[];
}

interface Modifier {
  id: string;
  name: string;           // "少冰"
  priceAdjustment: number; // 價格調整 (±)
  children?: ModifierGroup[]; // 嵌套子修改器（如「走甜」下面還有「全走/半走」）
}
```

**關鍵設計決策：** `maxSelect: 0` 表示無限制。這在東南亞的「加料」場景中很常見（加蝦、加肉、加蛋），但在歐美幾乎不會出現。

### 廚房顯示系統（KDS）的本地化

| 市場 | 廚房顯示需求 |
|------|-------------|
| 香港 | 中+英文、按單出菜、喊單模式 |
| 東南亞 | 多語言、清真/非清真分區 |
| 歐美 | 過敏原高亮、桌號管理 |
| 日本 | 精確到秒的出菜時間、手寫備註顯示 |

我們在KDS中設計了可配置的「標籤系統」：

- 🔴 過敏原提醒標籤
- 🟢 清真/非清真標籤
- 🟡 VIP/常客標籤
- 🔵 特殊要求標籤

每個市場可以配置要顯示哪些標籤。

## QR點餐：東南亞的殺手級應用

在香港，QR點餐還處於「可有可無」的階段，但在東南亞已經是標配。

**原因：** 東南亞的勞工成本相對低但增長快，加上疫情期間養成了QR掃碼習慣。在新加坡和泰國，超過60%的餐廳已經採用QR點餐。

我們的QR點餐產品設計原則：
1. **3秒進入點餐**：掃碼後不需要登錄就能點
2. **圖片為主**：東南亞用戶偏好看圖點餐
3. **支援團購模式**：一桌多人同時掃碼加單
4. **多語言自動切換**：根據手機系統語言

## 報表系統：不同市場關心不同的數字

| 市場 | 最關心的報表 |
|------|-------------|
| 香港 | 每單均價、翻桌率、外賣佔比 |
| 東南亞 | QR點餐佔比、現金vs電子支付比例 |
| 歐美 | 人工成本佔比、小費報表 |
| 中東 | 祈禱時間營業分析 |

報表系統不能一刀切，需要可配置的儀表板，讓餐廳選擇他們最關心的指標。

## 產品設計原則總結

1. **核心引擎全球化**：訂單、結帳、廚房顯示的底層邏輯不變
2. **配置層本地化**：菜單結構、稅率、支付方式做成可配置
3. **UI可換膚**：不只是翻譯，佈局和交互都要可定制
4. **預設不等於限制**：提供市場預設配置，但允許餐廳深度自定義
